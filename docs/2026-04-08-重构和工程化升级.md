## 原始 18 优化点实施详情

### P0 — 影响正确性（全部完成）

#### 优化点 1：批量 Ingest 并发竞态条件 ✅

- **方案**：使用 `Semaphore` 信号量 + `ioMutex` 互斥锁
- **改动**：
  - 新增 `src/shared/semaphore.ts` — 通用信号量实现
  - 重构 `workflows/ingest.ts` 的 `ingestBatch` — API 调用通过 `apiSemaphore` 控制并发，文件 I/O 和缓存更新通过 `ioMutex` 串行化
  - 废弃手动 worker 模式，改用 `Promise.allSettled` + 双层信号量

#### 优化点 2：LLM API 重试机制 ✅

- **方案**：封装 `callLlm` 函数，支持指数退避 + 抖动重试
- **改动**：
  - `shared/llm-utils.ts` — 新增 `callLlm()` 函数，支持 `maxRetries`、`baseDelayMs`、`maxDelayMs`、`timeoutMs` 配置
  - 可重试状态码：429, 500, 503, 529
  - 支持 `AbortController` 超时（signal 传入 SDK）
  - 4 个工作流全部替换 `client.messages.create` → `callLlm`

#### 优化点 4：API 响应验证（引入 zod）✅

- **方案**：使用 zod v4 进行运行时 schema 验证
- **改动**：
  - 新增 `src/shared/validators.ts` — `IngestResultSchema`、`InferredRelationSchema`、`PageSelectionSchema`
  - `workflows/ingest.ts` — 使用 `IngestResultSchema.safeParse` 替代手动 `isIngestResult`
  - `workflows/graph.ts` — 使用 `InferredRelationSchema.safeParse` 替代 `as Record<string, unknown>` 类型断言
  - `workflows/query.ts` — 使用 `PageSelectionSchema.safeParse` 验证页面选择结果

### P1 — 影响可靠性和可维护性（全部完成）

#### 优化点 3：原子文件写入 + 事务回滚 ✅

- **方案**：write-to-tmp + rename 模式 + `IngestTransaction` 事务类
- **改动**：
  - `shared/file-utils.ts` — `writeFile` 改为原子写入（tmp.pid.timestamp + rename），`appendLog` 同样原子化
  - `workflows/ingest.ts` — 新增 `IngestTransaction` 类，所有 wiki 写入操作先收集再批量 commit，失败自动 rollback

#### 优化点 7：sha256 函数统一 ✅

- **方案**：抽取为独立 `crypto-utils.ts` 模块
- **改动**：
  - 新增 `src/shared/crypto-utils.ts` — `sha256Full()` (64字符) 和 `sha256Short()` (16字符)
  - `shared/ingest-cache.ts` — 使用 `sha256Short`，保留兼容性 `@deprecated` 导出
  - `workflows/graph.ts` — 使用 `sha256Full`
  - 删除 `graph.ts` 中本地 `sha256` 定义和 `crypto` import

#### 优化点 8：lint fileCache 重构 ✅

- **方案**：消除模块级可变状态，改为参数传递
- **改动**：
  - `workflows/lint.ts` — 删除 `fileCache`、`readFileSync`、`preloadFiles`
  - `findOrphans`、`findBrokenLinks`、`findMissingEntities` 签名增加 `contents: Map<string, string>` 参数
  - `runLint` 使用 `readFilesParallel` 并行预加载后传入各函数

#### 优化点 11：结构化错误处理 ✅

- **方案**：`WikiError` + `WikiErrorCode` 枚举
- **改动**：
  - 新增 `src/shared/errors.ts` — `WikiError` 类，7 种错误码
  - `shared/file-utils.ts` — `readFile` 区分 ENOENT（返回空）和其他错误（抛 `WikiError`）
  - `workflows/ingest.ts` — `ingestSingle` 抛 `WikiError` 替代 `process.exit(1)`
  - `workflows/query.ts` — `query` 抛 `WikiError` 替代 `process.exit(1)`
  - 4 个 `cli/*.ts` — 统一捕获 `WikiError` 后 `process.exit(1)`

### P2 — 提升性能和体验（全部完成）

#### 优化点 5：文件 I/O 并行化 ✅

- **方案**：`readFilesParallel` + `buildPageNameMap` 接受外部 pages 参数
- **改动**：
  - `shared/file-utils.ts` — 新增 `readFilesParallel(paths, concurrency=20)` 批量并行读取
  - `shared/wiki-utils.ts` — `buildPageNameMap(pages?)` 支持可选参数，避免重复遍历
  - `workflows/lint.ts` — 使用 `readFilesParallel` + `buildPageNameMap(pages)`
  - `workflows/graph.ts` — 使用 `readFilesParallel` 替代串行读取

#### 优化点 6：Graph 推断阶段 API 并行化 ✅

- **方案**：`Semaphore` + `Promise.allSettled` 并发推断
- **改动**：
  - `workflows/graph.ts` — `buildInferredEdges` 新增 `concurrency=5` 参数
  - 使用 `Semaphore` 控制并发 API 调用数

#### 优化点 9：Token 用量追踪 ✅

- **方案**：全局 `UsageTracker` 单例 + 每次 API 调用自动记录
- **改动**：
  - 新增 `src/shared/usage-tracker.ts` — `usageTracker` 单例
  - `shared/llm-utils.ts` — `callLlm` 自动记录 usage
  - 4 个工作流末尾调用 `usageTracker.printSummary()`
  - 支持成本估算（Sonnet/Haiku 不同定价）

#### 优化点 13：Query 页面选择改进 ✅

- **方案**：tokenize + 评分排序
- **改动**：
  - `workflows/query.ts` — 重构 `findRelevantPages`
  - 新增 `tokenize()` 支持驼峰拆分
  - 不再过滤短词（保留 RAG/LLM/API 等缩写）
  - 评分系统：精确匹配 +3，部分匹配 +1，完整标题匹配 +5
  - 按分数降序排序后返回

#### 优化点 14：Context Window 管理 ✅

- **方案**：`fitPagesIntoBudget` 函数 + 段落边界截断
- **改动**：
  - 新增 `src/shared/context-manager.ts` — 通用上下文预算管理
  - `workflows/query.ts` — 读取页面内容后通过 `fitPagesIntoBudget` 限制总长度
  - `workflows/ingest.ts` — `buildWikiContext` 中使用截断策略
